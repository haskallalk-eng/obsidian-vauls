## 📝 Narrative

- **Root-Cause-Analyse „Recording läuft trotz Widerspruch weiter"**: drei gestapelte Bugs. (1) Prompt befahl `transfer_call` aber das registrierte Tool hieß `transfer_<sanitized>`; (2) `webhook_url` war auf dem Agent nicht gesetzt, deshalb kam `call_ended` nie → unsere DELETE-Logik feuerte nie; (3) `recording_declined` Tool + DB-Flag + Delete-Call-Pfad neu gebaut. Alle drei heute durch — erfolgreich mit einem Test-Call verifiziert (Retell zeigt den Call danach nicht mehr an).
- **Stripe-first Registrierung**: UX-Entscheidung, dass ein abgebrochener Checkout keinen halb-provisionierten Account hinterlassen darf. Neue Pending-Table ist idempotent zu beiden Success-Pfaden (Webhook + Frontend-Finalize). Beide führen über den gleichen `materializePendingFromSession()` Helper mit Transaktion + FOR UPDATE lock.
- **Auto-Apply raus**: User hat entschieden dass Chipy nie still Prompt ändern darf. Occurrence-Counter zählen weiter (für „5× aufgetreten"-UI-Hinweis), aber die Auto-Apply-Branch in `insights.ts` ist komplett deaktiviert. LLM-Guardrails verhindern dass jemand eine UI-Empfehlung („verbinde Google Calendar") als Prompt-Addition zurückbekommt.
- **Voice-Catalog Ehrlichkeits-Sweep**: nach Bug-Report, dass der „keine native Stimme"-Hinweis bei Ukrainisch/Arabisch/… nicht erscheint obwohl der Default `11labs-Marissa` dort nur via Multilingual-v2-Transfer spricht. Entscheidung im Gespräch: **Provider-Wahrheit > Auswahl-Größe**. HQ-Tier fliegt aus den meisten Non-EN-Sprachen raus (weil die 11labs-Actors englische Muttersprachler sind), bei FR/ES/IT/TR/PL/NL bleibt es bei den 9 Cartesia-Voices — ehrlich weniger, dafür klingen sie alle native. Für 15 Sprachen ohne jeden nativen Actor zeigen wir 3 Fallback-Stimmen plus Banner, das den Accent und den Voice-Clone-Ausweg erklärt. Der zusätzliche `few`-State (2 Cartesia-Voices für PT/RU/JA/KO/ZH/HI/SV) bekommt auch einen Hinweis — andere Formulierung: „nur wenige native Stimmen, für mehr Charakter → Clone". Ein separater Auto-Fallback fängt Legacy-Voice-IDs ab, die nach dem Sweep aus dem Catalog geflogen sind (`11labs-Carola` stand noch bei mir im Default-Agent), so dass der Voice-Picker-Trigger nicht plötzlich eine Supplier-ID zeigt.
- **Poll-Sync brachte Monate-alten stillen Produktionsbug ans Licht**: Google Calendar API war nie im Cloud-Projekt `145652898743` aktiviert. Der Agent-`freeBusy`-Call + jede `bookSlot`-Spiegelung nach Google hat seit Tag 1 still 403 gekriegt — `findSlotsForConnection` hatte zwei `catch {}` die den Fehler wegschluckten + `canCheckConnection` filterte die Connection dann aus der Liste → Agent arbeitete fröhlich mit Chipy-only weiter, 0 Log-Einträge. Die Bedingung war also seit Beginn: **„Google-connected" war de facto ein Placebo-Badge**, die Buchungen tauchten nie im externen Google-Kalender auf, Konflikt-Erkennung gegen bestehende Google-Termine fand nie statt. Mein neuer Poll-Sync war der erste Code-Pfad der den Fehler in `last_sync_error` schrieb statt zu catchen — sofort sichtbar, User konnte die API in 30 Sek aktivieren. Lehre für's nächste Mal: `catch {}` ist auch bei „defense in depth"-Fallbacks ein Anti-Pattern — mindestens `log.warn` mit dem error.message, sonst stapeln sich solche stillen Regressionen unbemerkt. Der Fix für beide silent-catches liegt in `d2e5901`.
- **Sync ist bewusst Display-only, nicht Agent-kritisch**: der Agent-freeBusy-Pfad bleibt authoritativ für Konflikt-Checks (live Google-API-Call pro Slot-Anfrage). Der neue Cache dient ausschließlich dem UI damit externe Events mit **Titeln** gerendert werden — freeBusy gibt aus Privacy-Gründen nur Busy-Zeiträume, keine Event-Titel. Wichtig: falls der Cache-Code komplett kaputt ist, arbeitet der Agent weiter korrekt. Design-Entscheidung im Kommentar von `calendar-sync.ts` oben dokumentiert so dass zukünftige Bug-Hunts nicht versehentlich den Agent-Pfad umleiten.
- **„24:00" war nicht der Bug, nur das Symptom**: der User meldete zwei verwandte Zeichen — Browser-Console-Warning `"24:00" does not conform to HH:mm` + 500 auf Save. Intuition wollte beides als ein Problem behandeln. Backend-Log zerlegte sie sauber: (1) das 24/7-Preset war UX-falsch (24:00 statt 23:59, reine HTML-Validation-Noise), (2) der 500er war ein Zod-Max-Length auf `services[].name`. Hätte ich (1) gefixt und auf „klappt jetzt"-Test gewartet, wäre (2) weiter unentdeckt geblieben — der Name-Bug reproduziert nur mit langem Paste. Backend-Logs-First ist die richtige Reihenfolge: „user-visible Symptom" ≠ „root cause", vor allem wenn zwei Fehlermeldungen gleichzeitig auf dem Schirm sind.
- **Skill-Bau als Meta-Arbeit**: entschieden einen eigenen Workflow-Skill zu bauen statt es wieder in Memory+CLAUDE.md zu streuen. Recherche vorher durchgezogen (Web-Search für 2025/2026-Obsidian-Best-Practices + Vault-Setup-Check) um zu verifizieren dass die Lücke real ist und nicht von den drei existierenden `obsidian-*`-Skills abgedeckt. Nebenbei: Skills-Ordner selbst unversioniert bis heute — private GitHub-Repo aufgesetzt, via SSH gepusht nachdem HTTPS stumm auf Auth-Prompt hing (SSH-Key aus Phonbot-Workflow war der richtige Weg). Lehre: bei `git push` Hangs immer erst `git remote -v` auf SSH-vs-HTTPS checken.
